> **4th place** out of 1,500+ participants in the **System Threat Forecaster** Kaggle competition.

## 🏆 Competition Overview

The goal of this competition is to predict a system's probability of getting infected by various families of malware, based on different properties of that system. The telemetry data containing these properties and the system infections was generated by threat reports collected by each system's antivirus software.

Each row in the dataset corresponds to a machine uniquely identified by a `MachineID`. The `target` column is the ground truth and indicates whether malware was detected on that machine. Using the information and labels in `train.csv`, the task is to predict the value of `target` for each machine in `test.csv`.

## 🔍 Exploratory Data Analysis (EDA)

Several targeted analyses were performed to understand the data and identify informative features:

1. **High-variance columns removed** — `MachineID` was dropped as it uniquely identifies individual samples and has no predictive correlation with the target.

2. **Low-variance column analysis** — Columns with very few unique values were examined by checking the distribution of the target label across each unique value:
   - `ProductName` was dropped because its values showed no significant difference in target distribution (both values yielded ~50% positive rate).
   - `DeviceFamily` was retained because `Windows.Server` systems showed a markedly different infection rate (~27.6%) compared to `Windows.Desktop` systems (~50.5%).

3. **Version string decomposition** — Columns such as `EngineVersion`, `AppVersion`, and `SignatureVersion` contained concatenated numeric values in string form. These were split into individual numeric sub-columns. ANOVA (Analysis of Variance) tests were then used to check for redundancy between newly extracted columns and existing numeric features — highly correlated pairs were pruned (p-value threshold: 0.05).

4. **Mutual information for categorical columns** — Mutual information scores were computed between pairs of categorical columns to identify redundant features. One column from each highly correlated pair was removed (e.g., `OSSkuFriendlyName` was dropped in favour of `OSEdition`; `OsPlatformSubRelease` was dropped in favour of `OSBranch`).

## ⚙️ Data Preparation

All preprocessing steps applied to `train.csv` were identically replicated on `test.csv` to ensure consistency.

| Step | Details |
|------|---------|
| **Null value imputation** | Mode imputation via `sklearn.impute.SimpleImputer` applied to all columns |
| **Column dropping** | Removed uninformative columns identified during EDA |
| **Version string splitting** | `EngineVersion`, `AppVersion`, `SignatureVersion`, `OSVersion`, `OSBuildLab` decomposed into numeric sub-columns |
| **Embedded numeric extraction** | Regex used to extract numeric values from categorical strings (e.g., `PlatformType`: `windows10` → `10`; `Processor` architecture suffix extracted) |
| **Date/time transformation** | `DateAS` and `DateOS` converted from `DD-MM-YYYY HH:MM` strings to minutes elapsed since the earliest date in the dataset, making them continuous numeric features |
| **`OSBuildLab` parsing** | Build number and build timestamp extracted as separate numeric features |
| **Dropped columns** | `MachineID`, `ProductName`, `OSArchitecture`, `OSSkuFriendlyName`, `NumericOSVersion`, `OsPlatformSubRelease` |

> **Motivation:** The majority of preprocessing steps were aimed at converting categorical features into numeric ones, which are universally more useful for classification models (excluding CatBoost). This also eliminated high-cardinality categorical columns, leaving only low-cardinality ones (max 28 unique values for `ChassisType`) suitable for One-Hot Encoding if needed.

## 🤖 Models

### 1. Logistic Regression
- **Preprocessing:** Z-score normalization for numeric features; Ordinal Encoding for categorical features
- **Tuning:** `GridSearchCV` over regularization strength `C` and `max_iter`
- **Best Accuracy:** `0.6128`

### 2. XGBoost ✅ *(Final Model)*
- **Preprocessing:** Ordinal Encoding for categorical features (consistent encoding across train/test using shared unique-value lists)
- **Tuning:** `GridSearchCV` over `n_estimators`, `max_depth`, `learning_rate`, `subsample`, `colsample_bytree`, `gamma`
- **Feature Selection:** Features ranked by XGBoost importance scores; top-k feature sweep revealed **top 59 features** gave the best accuracy
- **Best Hyperparameters:**

  | Parameter | Value |
  |-----------|-------|
  | `n_estimators` | 300 |
  | `max_depth` | 6 |
  | `learning_rate` | 0.1 |
  | `subsample` | 0.8 |
  | `colsample_bytree` | 0.8 |
  | `gamma` | 0.5 |

- **Best Accuracy: `0.6324`** 🏅

### 3. Random Forest
- **Preprocessing:** Ordinal Encoding (same strategy as XGBoost)
- **Tuning:** `GridSearchCV` over `n_estimators`, `max_depth`, `min_samples_split`, `min_samples_leaf`, `max_features`
- **Feature Selection:** Top-k feature sweep; **top 39 features** gave the best accuracy
- **Best Accuracy:** `0.6285`
